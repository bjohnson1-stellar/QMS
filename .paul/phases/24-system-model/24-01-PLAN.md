<objective>
## Goal
Build the equipment system model — a system type taxonomy, multi-system membership junction table, system consolidation engine that auto-detects logical systems from relationship data, backfill that assigns 77% of unassigned equipment to systems, and a "System" view on the dashboard showing the three-tier hierarchy (System → Equipment → Components).

## Purpose
Only 23% of equipment (107/465) has a system assignment. Refrigeration has 13 "systems" that are really compressor circuits (RS-1/RS-2/RS-3 all serve the same RAHUs). Plumbing has zero systems. The equipment dashboard needs meaningful system grouping to answer "what's in the medium-temp refrigeration system?" or "what equipment is on the domestic hot water loop?"

Research across all MEP disciplines confirmed:
- Refrigeration: 3-4 tier hierarchy (plant → system → suction group → equipment)
- HVAC: 2-3 tiers (air system → terminal units)
- Plumbing: 1-2 tiers (system type → equipment)
- Electrical: 4-5 tiers (service → switchboard → distribution → panel)
- The existing schema supports all of this via `parent_system_id` (unused until now)

## Output
- `pipeline/equipment_schema.sql` — New tables: `equipment_system_types`, `equipment_system_members`
- `pipeline/system_builder.py` — System consolidation and backfill engine
- `api/equipment.py` — New API endpoint for system data
- `frontend/templates/equipment/dashboard.html` — System view mode on dashboard
</objective>

  <action>
    **Create `pipeline/system_builder.py`** with these functions:

    **`consolidate_refrigeration(project_id, conn)`:**
    1. Query all RCU → RAHU "serves" relationships
    2. Group RCUs by the RAHU set they serve (RCUs serving identical sets = same logical system)
    3. For each group:
       - Create a parent system (REFRIG-1, REFRIG-2, etc.) with system_type_id = REFRIG
       - Set existing RS-* circuits as children via parent_system_id
       - Assign all RAHUs in the group to the parent system (UPDATE system_id)
       - Assign RAHU sub-components (parent_tag = RAHU-*) to same system
    4. Return count of systems created and equipment assigned

    Detection logic for grouping RCUs:
    ```python
    # Build map: rcu_tag → set of RAHU tags served (exclude sub-components)
    rcu_to_rahus = {}
    for rel in serves_relationships:
        if rel.target_tag is primary RAHU (no parent_tag):
            rcu_to_rahus.setdefault(rel.source_tag, set()).add(rel.target_tag)

    # Group RCUs by identical RAHU sets
    rahu_set_to_rcus = {}
    for rcu, rahus in rcu_to_rahus.items():
        key = frozenset(rahus)
        rahu_set_to_rcus.setdefault(key, []).append(rcu)

    # Each group = one consolidated system
    ```

    **`create_plumbing_systems(project_id, conn)`:**
    Create 5 systems and assign equipment by type:
    - PLMB-DHW: types Boiler, Water Heater, Storage Tank + pump tags P-H*
    - PLMB-DCW: types Water Softener, Backflow Preventer, Booster Pump + pump tags P-C*, P-D*
    - PLMB-CMP: types Compressor, Air Dryer, Air Receiver
    - PLMB-FIX: types Sink, Water Closet, Urinal, Shower, Mop Basin, Floor Sink, Water Cooler
    - PLMB-DRN: types Floor Drain, Floor Cleanout, Hub Drain

    **`fix_hvac_systems(project_id, conn)`:**
    1. Assign each AHU-N to its HVAC-AHU-N system (currently orphaned)
    2. Set system_type_id = HVAC-AIR on all HVAC-AHU-* systems
    3. Delete empty HVAC systems that have no members and no matching AHU

    **`assign_utility_instruments(project_id, conn)`:**
    For utility instruments (CV-01, FM-W1, etc.) that have no system, attempt to assign based on:
    - If tag references a known system equipment via relationships → assign to that system
    - Otherwise leave unassigned (will be caught by multi-pass extraction later)

    **`build_systems(project_id)` — orchestrator:**
    Runs all the above in order, returns summary dict with counts.

    Avoid:
    - Don't delete any existing RS-* systems — keep them as circuit-level children
    - Don't modify ELEC-SVC-* systems — they're already well-organized
    - Don't force-assign equipment when the system is ambiguous — leave for manual/Phase 25
    - Don't modify equipment_relationships — only read them for inference
