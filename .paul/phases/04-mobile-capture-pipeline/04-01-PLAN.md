<objective>
## Goal
Create the core mobile capture processing engine: scan a configured OneDrive folder for photos, analyze each with Claude vision API to extract structured quality observation data, and create quality_issues records with attached images.

## Purpose
Brandon's field workflow — take photos on phone, have them auto-sync to OneDrive, then processed into structured quality observations without manual data entry. This eliminates the ~10 minute manual entry time per observation and captures data in real-time while on site.

## Output
- `quality/mobile_capture.py` — processing engine (scan, analyze, create)
- `quality/cli.py` — `capture` subcommand with --dry-run, --folder, --project
- `config.yaml` — `mobile_capture:` config section
- `tests/test_mobile_capture.py` — unit tests with mocked Claude API
</objective>

  <name>Task 1: Create mobile capture processing module</name>
  <files>quality/mobile_capture.py, quality/schema.sql</files>
  <action>
    Create `quality/mobile_capture.py` with these components:

    **Schema addition (in quality/schema.sql):**
    Add `capture_log` table to track processed files:
    ```sql
    CREATE TABLE IF NOT EXISTS capture_log (
        id INTEGER PRIMARY KEY,
        filename TEXT NOT NULL,
        filepath TEXT NOT NULL,
        file_hash TEXT,
        status TEXT DEFAULT 'processed' CHECK (status IN ('processed', 'failed', 'skipped')),
        issue_id INTEGER REFERENCES quality_issues(id),
        error_message TEXT,
        processed_at TEXT DEFAULT CURRENT_TIMESTAMP,
        UNIQUE(filepath)
    );
    ```

    **Core functions in mobile_capture.py:**

    1. `scan_capture_folder(folder: Path, conn: sqlite3.Connection) -> List[Path]`
       - List all .jpg, .jpeg, .png, .heic files in folder (non-recursive)
       - Filter out files already in capture_log (by filepath)
       - Return list of new files to process

    2. `analyze_photo(image_path: Path, model: str = "sonnet") -> Dict`
       - Read image file, base64-encode it
       - Determine media type from extension (image/jpeg, image/png, image/heic)
       - Call `anthropic.Anthropic().messages.create()` with:
         - Model mapped from config name to API model ID (use existing model routing pattern)
         - System prompt: "You are a construction quality observation analyst..."
         - User message with image + structured extraction prompt
         - Request JSON response with: title, type, trade, severity, location, description, recommended_action
       - Parse JSON from response content
       - Normalize trade/type/status using existing quality/db.py helpers
       - Return dict with quality_issues-compatible fields
       - On API error: return dict with error field set

    3. `create_issue_from_capture(conn, analysis: Dict, image_path: Path, project_id: Optional[int], attachment_dir: Path) -> int`
       - INSERT into quality_issues with source='mobile', fields from analysis
       - Copy image file to attachment_dir/{issue_id}/
       - INSERT into quality_issue_attachments with file_type='image', filepath set
       - INSERT into capture_log with status='processed', issue_id set
       - Return issue_id

    4. `process_captures(folder: Path, project_id: Optional[int] = None, dry_run: bool = False) -> Dict`
       - Main orchestrator: scan → analyze each → create issues
       - Uses get_db() context manager for database operations
       - Reads config via get_config_value() for model, attachment_dir, default_project
       - Returns summary dict: {processed: N, failed: N, skipped: N, issues_created: [...]}
       - On dry_run: analyze but don't insert, log what would happen

    **Model ID mapping:**
    Use the same pattern as welding/extraction — config says "sonnet"/"opus"/"haiku", map to actual model IDs:
    - "sonnet" → "claude-sonnet-4-5-20250929"
    - "opus" → "claude-opus-4-6"
    - "haiku" → "claude-haiku-4-5-20251001"

    **Claude vision prompt design:**
    The system prompt should instruct Claude to analyze construction site photos and extract:
    - title: Short descriptive title for the observation
    - type: One of observation, ncr, deficiency, punch (default: observation)
    - trade: The responsible trade (Mechanical, Electrical, Plumbing, etc.)
    - severity: low, medium, high, critical
    - location: Where on site (describe what's visible)
    - description: Detailed description of what's observed
    - recommended_action: What should be done about it

    Response format: JSON object with these fields.

    **Avoid:**
    - Do NOT create a separate anthropic client wrapper — use `anthropic.Anthropic()` directly (matches existing pattern)
    - Do NOT add HEIC conversion — Claude API handles HEIC natively via base64
    - Do NOT use async — keep synchronous like existing extractors
    - Do NOT download from URLs — files are local OneDrive sync
