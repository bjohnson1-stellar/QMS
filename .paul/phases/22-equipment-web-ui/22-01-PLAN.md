<objective>
## Goal
Build the equipment web UI module — a dashboard with project stats, filterable equipment list, and a tabbed detail page showing equipment overview, cross-discipline connections, and conflicts/compliance. This completes the v0.4 Equipment-Centric Platform by putting all Phase 19-21 backend work behind a usable web interface.

## Purpose
Phases 19-21 built a powerful equipment backend (registry, conflict detection, spec compliance, impact chains) accessible only via CLI. Phase 22 makes this data accessible through the web UI, enabling visual exploration of equipment across projects, relationships, and conflicts — the primary user interface for day-to-day equipment coordination review.

## Output
- `api/equipment.py` — Flask blueprint with page routes and JSON API endpoints
- `frontend/templates/equipment/dashboard.html` — Project-aware dashboard with stats and equipment list
- `frontend/templates/equipment/detail.html` — Tabbed equipment detail page
- Config + blueprint registration for the equipment web module
</objective>

  <files>api/equipment.py, api/__init__.py, config.yaml, frontend/templates/equipment/dashboard.html</files>
  <action>
    **1. Add equipment to config.yaml web_modules:**
    ```yaml
    equipment:
      label: "Equipment"
      default_endpoint: "equipment.dashboard"
    ```
    Add after the `pipeline` entry (keeps related modules together).

    **2. Create api/equipment.py blueprint:**

    Page routes:
    - `GET /equipment/` → dashboard page (renders dashboard.html)
      - Query projects that have equipment_instances
      - Accept `?project=<number>` query param for project selection
      - Pass project list + selected project stats to template
    - `GET /equipment/<int:project_id>/<tag>` → detail page (renders detail.html)

    JSON API routes:
    - `GET /equipment/api/stats/<int:project_id>` → equipment stats (from get_equipment_stats)
    - `GET /equipment/api/list/<int:project_id>` → equipment list with filters
      - Query params: discipline, system_id, search (tag substring), lifecycle_stage
      - Join equipment_types for type_name, equipment_systems for system_tag
      - Return JSON array with: id, tag, type_name, discipline_primary, system_tag, lifecycle_stage, hp, voltage
    - `GET /equipment/api/conflicts/<int:project_id>` → conflict summary (from get_conflict_summary)
    - `GET /equipment/api/instance/<int:project_id>/<tag>` → full instance detail with appearances, relationships, conflicts

    Follow existing blueprint patterns from api/pipeline.py and api/licenses.py:
    - `bp = Blueprint("equipment", __name__, url_prefix="/equipment")`
    - Use `@module_required("equipment")` decorator if available, otherwise just the auth gate handles it
    - Lazy imports inside route handlers (consistent with existing patterns)

    **3. Register blueprint in api/__init__.py:**
    Add after the pipeline blueprint registration:
    ```python
    from qms.api.equipment import bp as equipment_bp
    app.register_blueprint(equipment_bp)
    ```

    **4. Create frontend/templates/equipment/dashboard.html:**
    Follow existing UI patterns (vanilla JS, fetch API, stats-grid, card layout):

    - Page header: "Equipment Registry" with project selector dropdown
    - Stats grid (6 cards): Total Equipment, Types, Systems, Relationships, Conflicts (critical count), Spec Violations
    - Conflict summary card: counts by type (attribute/missing_discipline/spec_violation) and severity, styled with color coding
    - Filter bar: discipline dropdown + text search input + lifecycle stage dropdown
    - Equipment table: Tag (link to detail), Type, Discipline, System, Stage (color badge), HP, Voltage
    - Table rows clickable → navigate to detail page
    - Project selector: dropdown of projects with equipment data, auto-loads on change
    - Loading states and empty states

    JavaScript pattern (inline, consistent with other modules):
    ```javascript
    let currentProject = null;
    function loadDashboard(projectId) {
      currentProject = projectId;
      Promise.all([
        fetch(`/equipment/api/stats/${projectId}`).then(r => r.json()),
        fetch(`/equipment/api/list/${projectId}`).then(r => r.json()),
        fetch(`/equipment/api/conflicts/${projectId}`).then(r => r.json()),
      ]).then(([stats, equipment, conflicts]) => {
        renderStats(stats, conflicts);
        renderTable(equipment);
        renderConflictSummary(conflicts);
      });
    }
    ```

    CSS: Use existing style.css classes (stats-grid, stat-card, card, table styles). Add minimal inline styles only for equipment-specific elements (lifecycle stage badges).

    Avoid:
    - Don't add jQuery or external JS libraries — vanilla JS only
    - Don't create a separate CSS file — use existing style.css classes + minimal inline styles
    - Don't add authentication decorators beyond what the auth gate provides
  </action>
