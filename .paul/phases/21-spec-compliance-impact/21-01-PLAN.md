<objective>
## Goal
Build a spec compliance checking engine and equipment impact chain analyzer for the equipment-centric platform. The compliance engine validates equipment attributes against defined requirements, generating spec_violation conflicts. The impact analyzer traverses the equipment relationship graph to trace upstream/downstream dependencies and assess blast radius of changes or violations.

## Purpose
Phases 19-20 built the equipment registry and cross-discipline conflict detection. Phase 21 adds the next intelligence layer: verifying equipment meets specification requirements (not just cross-discipline consistency) and understanding how equipment connects through the relationship graph. This enables answering questions like "Does this motor meet the project voltage standard?" and "If this panel changes, what downstream equipment is affected?"

## Output
- `pipeline/spec_checker.py` — Spec compliance checking module
- `pipeline/impact_analyzer.py` — Equipment relationship graph traversal and impact analysis
- `equipment_spec_requirements` table with seeded Vital requirements
- Extended CLI commands: `spec-check` and `impact`
- Spec violation conflicts in `equipment_conflicts` table
</objective>

  <name>Task 1: Spec Compliance Engine — Schema, Checker, and CLI</name>
  <files>pipeline/equipment_schema.sql, pipeline/spec_checker.py, pipeline/cli.py, core/db.py</files>
  <action>
    **1. Add equipment_spec_requirements table to equipment_schema.sql:**
    ```sql
    CREATE TABLE IF NOT EXISTS equipment_spec_requirements (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        type_id INTEGER REFERENCES equipment_types(id),
        type_name TEXT,                          -- fallback when type_id not matched (e.g., "Switchboard")
        attribute_name TEXT NOT NULL,            -- attribute to check (voltage, phases, aic_rating, etc.)
        check_type TEXT NOT NULL DEFAULT 'exact', -- exact | min | max | range | one_of | regex
        expected_value TEXT NOT NULL,            -- expected value or JSON for complex checks
        severity TEXT NOT NULL DEFAULT 'warning', -- critical | warning | info
        source_spec TEXT,                        -- spec reference (e.g., "240000 Section 2.1")
        description TEXT,                        -- human-readable requirement description
        active INTEGER NOT NULL DEFAULT 1,
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    );
    CREATE INDEX IF NOT EXISTS idx_spec_req_type ON equipment_spec_requirements(type_id);
    CREATE INDEX IF NOT EXISTS idx_spec_req_name ON equipment_spec_requirements(type_name);
    ```
    - type_id is nullable — use type_name as fallback for broad rules (e.g., all equipment)
    - check_type "one_of" uses JSON array in expected_value: '["480V","480/277V"]'
    - check_type "min"/"max" uses numeric expected_value with tolerance

    **2. Seed requirements for Vital project equipment:**
    Insert via Python migration function (not raw SQL in schema file):
    - Switchboards: voltage one_of ["480V","480/277V"] (critical)
    - Distribution Panels: voltage one_of ["480V","480/277V"] (critical)
    - Panelboards: voltage one_of ["208/120V","480/277V"] (warning — depends on purpose)
    - Transformers: phases exact "3" (warning)
    - All electrical panels: aic_rating min 10000 (warning, "Minimum fault current rating")
    - Motors: voltage one_of ["460V","480V"] (warning, nameplate vs system)
    - Condensing Units: refrigerant exact "R-717" (critical, ammonia system)
    - Air Handling Units (RAHU): refrigerant exact "R-717" (critical)

    **3. Create pipeline/spec_checker.py module:**
    - `check_compliance(db, project_id)` → runs all active requirements against equipment instances
    - For each requirement:
      - Find matching equipment instances by type_id or type_name
      - Extract attribute value from equipment_appearances.attributes_on_sheet (JSON)
      - Also check equipment_instances direct columns (hp, voltage, amperage)
      - Apply check_type comparison:
        - exact: normalize and compare strings
        - min/max: parse numeric, compare
        - one_of: check membership in JSON array
        - range: parse "[min,max]" from expected_value
        - regex: re.match against value
      - Reuse voltage normalization from conflict_detector.py (_normalize_value, _parse_numeric)
    - Store violations in equipment_conflicts with conflict_type='spec_violation':
      - discipline_a = 'Spec', value_a = expected_value
      - discipline_b = discipline from appearance, value_b = actual_value
      - attribute_name = requirement attribute_name
      - rule_id = NULL (spec requirements are separate from conflict_rules)
    - Clear existing spec_violation conflicts before re-run (idempotent, same pattern as conflict_detector)
    - `get_compliance_summary(db, project_id)` → {total_checked, violations, by_severity, by_type}
    - `get_violations(db, project_id, severity=None, type_name=None)` → list of violation dicts

    **4. Add spec-check CLI command to pipeline/cli.py:**
    - `qms pipeline spec-check <project>` — runs compliance check and shows summary
    - Options: `--seed` to re-seed default requirements, `--severity` filter
    - Summary format similar to conflicts command (counts, severity breakdown)
    - Reuse existing project lookup pattern (number column, not project_number)

    **5. Wire migration in core/db.py:**
    - Add equipment_spec_requirements table creation to equipment schema migration
    - Add seed function call after table creation (idempotent INSERT OR IGNORE)

    Avoid:
    - Don't duplicate normalization logic from conflict_detector.py — import and reuse
    - Don't modify existing conflict_rules table — spec requirements are a separate concept
    - Don't add equipment attribute columns to equipment_instances — use existing JSON attributes

<task type="auto">
  <name>Task 2: Impact Chain Analyzer — Graph Traversal, Drawing Impact, and CLI</name>
  <files>pipeline/impact_analyzer.py, pipeline/cli.py</files>
  <action>
    **1. Create pipeline/impact_analyzer.py module:**

    **Forward impact (downstream):**
    - `get_forward_impact(db, project_id, tag, max_depth=10)` → list of {tag, type_name, depth, path, relationship_type}
    - BFS through equipment_relationships: source_tag → target_tag for "feeds", target_tag → source_tag for "serves" (reversed direction: if RCU serves RAHUs, then RCU change impacts RAHUs)
    - Actually: for "feeds" relationship (panel feeds motor), downstream = target. For "serves" relationship (RCU serves RAHU), downstream = target too.
    - Track visited set to prevent cycles
    - Return results ordered by depth, then tag

    **Reverse trace (upstream):**
    - `get_reverse_trace(db, project_id, tag, max_depth=10)` → list of {tag, type_name, depth, path, relationship_type}
    - BFS in reverse: target_tag → source_tag for "feeds", source_tag → target_tag for... actually:
      - "feeds": panel feeds equipment → upstream of equipment is panel → follow target_tag back to source_tag
      - "serves": RCU serves RAHU → upstream of RAHU is RCU → follow target_tag back to source_tag
    - Both relationship types: upstream = source_tag when current tag is target_tag

    **Drawing impact:**
    - `get_drawing_impact(db, project_id, drawing_number)` → {direct: [...equipment on drawing], indirect: [...downstream of direct], summary: {direct_count, indirect_count, total}}
    - Query equipment_appearances for the drawing_number
    - For each equipment on drawing, run forward impact
    - Deduplicate across chains
    - Return combined direct + indirect impact

    **Conflict propagation:**
    - `get_violation_impact(db, project_id)` → for each spec_violation conflict, trace downstream to see what else is affected
    - E.g., if panel 1HMSB has a voltage violation, and it feeds CH-1, then CH-1 inherits concern
    - Don't create new conflict records — return as analysis results (advisory, not authoritative)

    **2. Add impact CLI command to pipeline/cli.py:**
    - `qms pipeline impact <project> <tag>` — shows forward + reverse impact chains
    - `qms pipeline impact <project> --drawing <drawing_number>` — shows drawing revision impact
    - `qms pipeline impact <project> --violations` — shows violation propagation analysis
    - Format: indented tree showing depth levels
      ```
      1HMSB (Switchboard)
      ├── CH-1 (Chiller) [feeds, depth=1]
      ├── P-C1 (Pump) [feeds, depth=1]
      │   └── (no downstream)
      └── P-C2 (Pump) [feeds, depth=1]
      ```

    Avoid:
    - Don't modify equipment_conflicts for propagated impacts — these are advisory analysis, not authoritative conflicts
    - Don't traverse deeper than max_depth (default 10) — prevent runaway on circular data
    - Don't assume relationship direction — check both source_tag and target_tag appropriately
  </action>
  <verify>

    Run `qms pipeline impact 07645 --drawing E-401` — should show direct + indirect equipment impact.
    Run `qms pipeline impact 07645 RCU-1` — should show RAHU units served by RCU-1.
    Verify no infinite loops on circular relationships.
  </verify>
