<objective>
## Goal
Build equipment tag hierarchy — a tag parser that identifies parent-child relationships, a migration adding `parent_tag` and `component_type` columns, a backfill that deduplicates 124 reversed tag pairs and reclassifies sub-components, and a grouped dashboard view that reduces visual noise from 589 to ~276 primary equipment rows.

## Purpose
The equipment dashboard currently shows 589 flat items for Vital. Research revealed:
- **124 duplicate pairs**: `RAHU-2-CV2` (typed "Refrigeration AHU") and `CV2-SYS-RAHU-2` (typed "Instrument") are the same physical control valve
- **Sub-components treated as top-level**: valves (CV), pressure transmitters (PT), temperature transmitters (TT), strainers (STR) are interleaved with primary equipment
- **Wrong type classification**: RAHU-2-CV2 is classified as "Refrigeration AHU" instead of "Control Valve"
- **No hierarchy**: no way to see that RAHU-2 has 4 sub-components (CV2, PT3, TT4, STR1)

This plan makes the dashboard usable by grouping sub-components under their parent equipment and eliminating duplicates.

## Output
- `pipeline/tag_parser.py` — Tag parsing and normalization module
- Migration adding `parent_tag`, `component_type` columns to `equipment_instances`
- Backfill script deduplicating 124 reversed pairs, setting parent_tag for 124 sub-components
- Dashboard grouped view toggle (All | Grouped) using existing workforce/projects UI patterns
</objective>

  <files>pipeline/tag_parser.py, pipeline/equipment_schema.sql, core/db.py</files>
  <action>
    **1. Create `pipeline/tag_parser.py`:**

    Module with two core functions:

    `parse_tag(tag, existing_tags)` → Returns dict:
    ```python
    {
        "original": "RAHU-2-CV2",
        "parent_tag": "RAHU-2",       # or None if primary
        "component_code": "CV2",       # or None if primary
        "component_type": "Control Valve",  # or None if primary
        "is_reversed": False,
        "normalized": "RAHU-2-CV2",
    }
    ```

    Tag parsing logic:
    - **Forward sub-component** (`RAHU-2-CV2`): Match `{parent}-{suffix}` where parent exists in `existing_tags` and suffix matches a known component pattern. Use the LONGEST matching parent (e.g., for `RAHU-20-CV2`, parent is `RAHU-20` not `RAHU-2`).
    - **Reversed instrument** (`CV2-SYS-RAHU-2`): Match `{suffix}-SYS-{parent}` or `{suffix}-{parent}-SYS` patterns. Normalize to forward form.
    - **Standalone instrument** (`CV-01`): No parent found — leave as primary.
    - **Primary equipment** (`RAHU-2`, `RCU-1`, `1HMSB`): No suffix match — leave as primary.

    Component code → type mapping:
    ```python
    COMPONENT_TYPES = {
        "CV": "Control Valve",
        "PT": "Pressure Transmitter",
        "TT": "Temperature Transmitter",
        "STR": "Strainer",
        "SV": "Solenoid Valve",
        "FV": "Flow Valve",
        "PRV": "Pressure Relief Valve",
        "FM": "Flow Meter",
    }
    ```
    Extract the alpha prefix from the suffix (e.g., "CV2" → "CV", "STR1" → "STR") to look up the type.

    `normalize_reversed_tag(tag)` → Returns normalized forward form or original if not reversed:
    - `CV2-SYS-RAHU-2` → `RAHU-2-CV2`
    - `PT3-SYS-RAHU-20` → `RAHU-20-PT3`
    - `CV5-RAHU-20-SYS` → `RAHU-20-CV5`

  <name>Task 2: Backfill Parent Tags, Deduplicate Reversed Tags, Reclassify Types</name>
  <files>pipeline/tag_parser.py, pipeline/reconciler.py</files>
  <action>
    **1. Add `backfill_hierarchy(project_id)` function to `pipeline/tag_parser.py`:**

    Steps:
    a. Query all tags for the project: `SELECT tag FROM equipment_instances WHERE project_id = ?`
    b. Build set of all tags for parent lookup
    c. Run `parse_tag()` on each tag
    d. For each sub-component (parent_tag is not None):
       - UPDATE parent_tag and component_type on the record
    e. Identify duplicate pairs: find reversed tags where a matching forward tag exists
       - For each duplicate pair, merge any unique data from the reversed record (appearances, relationships, attributes) into the forward record
       - DELETE the reversed record
    f. Reclassify sub-component types:
       - For forward sub-components that were wrongly typed (e.g., "Refrigeration AHU" for a CV2), look up or create the correct equipment_type (e.g., "Control Valve")
       - UPDATE type_id on the equipment_instance

    **Duplicate merge logic** (for each reversed tag being deleted):
    - Move any `equipment_appearances` rows from reversed instance_id to forward instance_id (skip if duplicate UNIQUE constraint)
    - Move any `equipment_relationships` referencing the reversed tag to use the forward tag
    - Move any `equipment_conflicts` referencing the reversed tag to use the forward tag
    - Then DELETE the reversed equipment_instance

    **2. Add CLI command or integrate into reconciler:**

    Add a `backfill_hierarchy` call to `pipeline/reconciler.py` that can be triggered after reconciliation, or add a standalone function callable from the equipment API.

    Alternatively, add a route `POST /equipment/api/backfill/<int:project_id>` in `api/equipment.py` that triggers the backfill (admin-only).

    Avoid:
    - Don't delete any primary equipment records
    - Don't modify equipment that isn't a clear reversed duplicate (only delete when both forward and reversed exist)
    - Don't touch equipment_conflicts status — just update the tag reference
    - Preserve all appearance and relationship data during merges
