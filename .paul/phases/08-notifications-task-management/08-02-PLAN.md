<objective>
## Goal
Add Teams webhook delivery for urgent/critical license notifications, notification web API routes, and a task queue UI panel on the licenses dashboard — completing the notification & task management phase.

## Purpose
Plan 08-01 built the notification backend (rules, generators, CLI). This plan makes notifications visible and actionable: web UI for reviewing/acknowledging/resolving alerts, and Teams webhook push for urgent items so critical expirations aren't missed.

## Output
- Teams webhook sender in `licenses/notifications.py` (outgoing POST to Teams incoming webhook URL)
- 5 notification API endpoints in `api/licenses.py`
- Task queue panel on licenses dashboard with notification list, acknowledge/resolve actions
- Config entry for Teams webhook URL in `config.yaml`
</objective>

<acceptance_criteria>

## AC-1: Teams Webhook Delivery
```gherkin
Given a Teams incoming webhook URL is configured in config.yaml
When generate_all_notifications creates urgent or high-priority notifications
Then a formatted Teams message card is POSTed to the webhook URL
And the notification is marked as webhook_sent (or failure is logged)
```

## AC-2: Notification API Endpoints
```gherkin
Given the licenses blueprint is registered
When a user calls GET /licenses/api/notifications
Then active notifications are returned as JSON sorted by priority then due_date
And when POST /licenses/api/notifications/<id>/acknowledge is called
Then the notification status changes to 'acknowledged'
And when POST /licenses/api/notifications/<id>/resolve is called
Then the notification status changes to 'resolved'
```

## AC-3: Task Queue UI Panel
```gherkin
Given the licenses dashboard page loads
When there are active notifications in the database
Then a "Notifications" panel displays notification cards grouped by priority
And each card shows title, due_date, days_until_due, priority badge
And acknowledge/resolve action buttons work via fetch() calls to the API
```

## AC-4: Webhook Configuration
```gherkin
Given config.yaml has a licenses.teams_webhook_url entry
When the webhook URL is empty or missing
Then webhook sending is silently skipped (no errors)
And when the URL is present but the POST fails
Then the failure is logged but does not block notification generation
```

  <action>
    **In config.yaml:**
    Add under `licenses:` section (create if needed):
    ```yaml
    licenses:
      teams_webhook_url: ""   # Teams incoming webhook URL (leave empty to disable)
    ```

    **In licenses/notifications.py — add Teams webhook function:**
    - `send_teams_webhook(notifications: List[Dict]) -> Dict[str, int]`
    - POST to Teams incoming webhook URL using `urllib.request` (no new dependency)
    - Format as Teams Adaptive Card with: title, priority badge (color-coded), due date, days remaining
    - Group multiple notifications into a single card (max 10 per message)
    - Read webhook URL from config.yaml via `get_config()` → `licenses.teams_webhook_url`
    - Return {"sent": N, "failed": N}
    - If URL empty/missing, return {"sent": 0, "skipped": True} immediately
    - Catch all HTTP errors — log warning but never raise
    - Add optional `send_webhook=False` param to `generate_all_notifications()` — when True, sends webhook for newly created urgent/high notifications

    **In api/licenses.py — add 5 notification API routes:**
    1. `GET /api/notifications` → list_active_notifications(conn, limit=50) + get_notification_summary(conn)
    2. `GET /api/notifications/summary` → get_notification_summary(conn) only
    3. `POST /api/notifications/<int:notification_id>/acknowledge` → acknowledge_notification(conn, id, user)
    4. `POST /api/notifications/<int:notification_id>/resolve` → resolve_notification(conn, id, user)
    5. `POST /api/notifications/generate` → generate_all_notifications(conn) (admin only)

    All routes require auth. Generate route requires admin role.
    Follow existing api/licenses.py patterns: try/except, conn from get_db(), jsonify responses.
    Avoid: Creating a separate blueprint — add to existing licenses blueprint.
  </action>
