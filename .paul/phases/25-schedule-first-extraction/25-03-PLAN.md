<objective>
## Goal
Complete Vital MEP schedule extraction — process remaining MEP schedule sheets using Docling (IBM TableFormer) primary + discipline-specific Claude vision agents for gap-fill, with Opus shadow review for quality assurance. Re-extract the initial 5 sheets for consistency. Track extraction source, confidence tiers, and page numbers for traceability.

## Purpose
Plan 25-02 validated the harness with 5 sheets (203 entries) using pdfplumber agents. Testing revealed:
- **PyMuPDF find_tables()**: Fast (0.1s/page) but produces jumbled, unusable output on CAD-generated schedule drawings
- **Docling**: Slower (47s/page) but produces clean structured DataFrames with proper column mapping and merged cell handling
- **Claude vision**: Excellent contextual understanding, catches tables Docling misses

The optimal approach: Docling extracts structured tables (high accuracy, proper columns), Claude vision agents handle sheets where Docling misses tables or has low coverage. Both run locally/via subscription — zero additional cost.

**Scope narrowed to MEP only**: 34 Architectural schedule sheets (doors, windows, finishes) are excluded — they don't feed conflict detection, spec compliance, or equipment coordination. MEP target: 20 remaining + 5 re-extract = 25 sheets.

## Output
- `pipeline/docling_extractor.py` — Docling-based table extraction with discipline-specific prompts
- `schedule_extractions` schema additions: `page_number`, updated `extraction_model` usage
- All MEP schedule sheets processed with source tracking and tiered confidence
- Extraction quality report with shadow review metrics
</objective>

<task type="auto">
  <name>Task 2: Docling Extractor with Discipline-Specific Prompts</name>
  <files>pipeline/docling_extractor.py</files>
  <action>
    **Create `pipeline/docling_extractor.py`:**

    **`extract_tables_from_pdf(file_path)`** — Docling wrapper:
    - Initialize DocumentConverter once at module level (lazy, reuse across calls)
    - Convert PDF, iterate tables
    - Return list of dicts: {table_index, page_number, headers, rows (list of dicts), row_count, col_count}
    - Import docling inside function (optional dependency, graceful error message)

    **`tables_to_equipment(tables, discipline)`** — Column mapper:
    - Case-insensitive fuzzy column matching:
      - MARK/TAG/UNIT/ITEM → tag
      - AREA SERVED/DESCRIPTION/SERVICE → description
      - HP/HP EACH/MOTOR HP → hp
      - V/PH/HZ/VOLTAGE/ELECTRICAL → voltage (extract voltage value)
      - MANUFACTURER/MFR/BASIS OF DESIGN.MANUFACTURER → manufacturer
      - MODEL/MODEL NUMBER/BASIS OF DESIGN.MODEL → model_number
      - WEIGHT/WEIGHT (LBS)/OPERATING WEIGHT → weight_lbs
      - CFM/AIR FLOW/AIR FLOW.CFM → cfm
      - KW/KVA/CAPACITY (KW)/HEATING CAPACITY → kva
      - MCA/AMPERAGE/FLA → amperage
      - PHASE/PH → phase_count
      - CIRCUIT/CKT → circuit
      - PANEL/FED FROM/PANEL SOURCE → panel_source
    - Skip rows where tag column is empty or None
    - Include page_number from the table metadata
    - Return list of equipment dicts ready for store_schedule_data()

    **`get_discipline_prompt(discipline)`** — Returns discipline-specific extraction prompt for Claude vision agents:

    - **Electrical**: "Extract panel schedules. For each panel: tag, voltage, phase, amperage, AIC rating, fed-from. For each circuit load: tag, HP/kVA, breaker amps, starter type (FVNR/VFD), circuit number, panel source."
    - **Refrigeration**: "Extract AHU and condensing unit schedules. For each unit: tag, model, compressor HP, tonnage/capacity TR, refrigerant type, suction temp, fan count, fan HP, voltage, weight."
    - **Mechanical**: "Extract HVAC equipment schedules (fans, heaters, RTUs, ERVs, CRAC units). For each: tag, description, CFM, HP, voltage/phase, weight, manufacturer, model."
    - **Utility**: "Extract utility equipment schedules (pumps, chillers, boilers, compressors, water heaters, tanks). For each: tag, description, HP, flow GPM, head ft, voltage, manufacturer, model, weight."
    - **Plumbing**: "Extract plumbing fixture and pump schedules. For each: tag, description, type, manufacturer, model, connection sizes."
    - **Structural**: "Extract structural equipment schedules if present. For each: tag, description, capacity, manufacturer, model."
    - **Default**: Generic schedule extraction prompt

    Avoid:
    - Do NOT modify extraction_harness.py or schedule_extractor.py (done in Task 1)
    - Do NOT add CLI commands
    - Keep Docling import inside function (optional dependency)
  </action>

  <name>Task 4: Opus Shadow Review (10% Sample Verification)</name>
  <files>pipeline/extraction_harness.py</files>
  <action>
    **Independent verification of extraction accuracy using Opus shadow agents.**

    Select a 10% sample (~3 sheets) covering diverse disciplines and complexity levels:
    - 1 dense multi-schedule sheet (M6001 — ~12 schedules on one page)
    - 1 electrical panel schedule (E6102 — nested panel hierarchy)
    - 1 sheet that was gap-filled by Claude Sonnet in Step 3

    For each sampled sheet:
    1. Launch an **Opus agent** to independently read the PDF and extract all equipment
       - Agent has NO access to the primary extraction results
       - Agent uses the discipline-specific prompt from get_discipline_prompt()
       - This is a fresh, blind extraction
    2. Compare Opus extraction against primary (Docling/Sonnet) results:
       - **Missing tags**: tags Opus found that primary missed
       - **Extra tags**: tags in primary that Opus didn't find (potential hallucinations)
       - **Value mismatches**: same tag but different HP, voltage, manufacturer, etc.
    3. For each discrepancy:
       - If Opus is clearly correct (supported by drawing content): fix primary data
       - If ambiguous: flag for human review at checkpoint
       - If primary is correct and Opus hallucinated: note as false positive
    4. Corrected entries stored with model_used="claude-opus-shadow", confidence=0.98
    5. Record shadow review results:
       - Per-sheet: total entries, matches, discrepancies, discrepancy rate
       - Overall: aggregate accuracy metrics

    **Key principle**: Shadow agents are BLIND — they don't see the existing extraction.
    This prevents confirmation bias.

    **Corrections**: Use clear_schedule_data(project_id, sheet_id) then re-store corrected entries
    with appropriate model_used for each entry's source.
  </action>
