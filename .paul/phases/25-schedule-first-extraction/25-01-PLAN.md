<objective>
## Goal
Build the schedule-first extraction pipeline infrastructure — an extraction order engine that processes schedules before floor plans, a Haiku-powered schedule extractor specialized for tabular equipment data, and a context builder that creates per-sheet equipment checklists from schedule results for informed floor plan extraction.

## Purpose
Current extraction is single-pass and order-blind — every sheet gets the same treatment regardless of drawing type. This causes:
- **Hallucination**: Floor plan extraction fabricates equipment from schedules/legends (PP1114 hallucinated compressed air equipment)
- **Missed cross-discipline connections**: Electrical panel schedules reference RAHU-1 but extraction doesn't know what it is
- **Wrong discipline assignment**: Equipment gets assigned to whatever drawing it was first extracted from, not where it actually belongs
- **82.8% system assignment but only ~40% attribute completeness**

Schedule-first extraction solves this by building an authoritative equipment manifest from structured schedule data (tabular, Haiku-friendly) before processing spatial floor plans (complex, Sonnet-needed). Floor plan extraction then uses the manifest as a checklist — "which of these known tags appear here?" instead of "what can you find?"

## Output
- `pipeline/extraction_order.py` — Determines processing order: schedules → legends → plans/P&IDs
- `pipeline/schedule_extractor.py` — Haiku-based schedule extraction (tabular → equipment registry)
- `pipeline/context_builder.py` — Builds per-sheet equipment context for informed extraction
- Updated `pipeline/extractor.py` — Integrates context into extraction prompts
</objective>

## AC-1: Extraction Order Engine Determines Correct Processing Sequence
```gherkin
Given a project with 400+ sheets of mixed types
When the extraction order engine runs
Then it returns sheets grouped into phases: schedules first, then legends, then plans/P&IDs
And schedule sheets are identified by drawing_category='schedule' OR drawing_number matching x6xxx pattern
And legend sheets are identified by drawing_category in ('cover','legend') OR drawing_number matching x0001/x0002
And each phase has the correct model assignment (Haiku for schedules/legends, Sonnet for plans)
```

## AC-2: Schedule Extractor Produces Structured Equipment Data from Tabular Drawings
```gherkin
Given an electrical panel schedule drawing (E6102)
When the schedule extractor processes it with Haiku
Then it returns structured equipment data: tag, description, hp/kva, voltage, circuit, panel source
And the data is stored in a schedule_extractions staging table
And equipment tags match the format found on the actual drawings
And no hallucinated equipment is created (only what's in the table on the drawing)
```

## AC-3: Context Builder Creates Equipment Checklist for Floor Plan Extraction
```gherkin
Given schedule extraction has completed for a project
When the context builder generates context for a utility floor plan drawing
Then it returns a focused equipment checklist filtered by discipline and floor
And the checklist includes: tag, type, description, expected attributes
And the floor plan extraction prompt includes "find these specific tags" instead of open-ended search
```

## AC-4: Informed Extraction Uses Context to Reduce Hallucination
```gherkin
Given the context builder has produced a checklist for a floor plan
When the informed extraction processes the floor plan with Sonnet
Then it reports which checklist items were found and their locations
And it flags any equipment found that is NOT on the checklist (potential new discovery OR hallucination)
And found-on-checklist items get high confidence, off-checklist items get low confidence for review
```

<task type="auto">
  <name>Task 2: Schedule Extractor — Haiku-Based Tabular Extraction</name>
  <files>pipeline/schedule_extractor.py</files>
  <action>
    **Create `pipeline/schedule_extractor.py`:**

    `extract_schedule(sheet_id, project_id)` → Processes a single schedule drawing:
    1. Load PDF as image (Claude vision) or extract text (PyMuPDF)
    2. Build a Haiku prompt specialized for tabular schedule data:
       ```
       You are extracting equipment data from an engineering schedule drawing.
       This is a {discipline} schedule drawing: {file_name}

       Extract ALL equipment entries from the schedule table(s). For each entry return:
       - tag: Equipment tag/identifier (exactly as shown)
       - description: Equipment description
       - equipment_type: Type of equipment (e.g., "Distribution Panel", "Pump", "Air Compressor")
       - hp: Horsepower (if shown)
       - kva: KVA rating (if shown)
       - voltage: Voltage (if shown)
       - amperage: Amperage/FLA (if shown)
       - circuit: Circuit number or breaker (if shown, for panel schedules)
       - panel_source: Panel/source feeding this equipment (if shown)
       - manufacturer: Manufacturer (if shown)
       - model_number: Model number (if shown)
       - weight_lbs: Weight (if shown)
       - cfm: Airflow CFM (if shown, for HVAC schedules)

       Return as JSON array. Only extract what is explicitly shown in the table.
       Do NOT infer or fabricate data that is not visible.
       ```
    3. Call Haiku via Claude API (follow existing pattern from refrigeration_extractor.py)
    4. Parse JSON response
    5. Store results in schedule_extractions table

    `extract_all_schedules(project_id)` → Orchestrator:
    1. Get schedule sheets from extraction_order
    2. For each sheet, call extract_schedule()
    3. Return summary: sheets processed, equipment found, errors

    **Model selection**: Always use Haiku for schedules (structured tabular data). Use Claude vision if the PDF has table graphics that PyMuPDF can't parse as text.

    **Detection of vision vs text**: If `extract_pdf_text()` returns < 100 chars for a schedule sheet, it's likely a scanned/graphical schedule → use vision. Otherwise use text-based extraction (cheaper).

    Follow patterns from existing extractors:
    - `refrigeration_extractor.py` for Claude API call pattern
    - `extractor.py:extract_pdf_text()` for text extraction
    - Error handling: log and continue on per-sheet failures

    Avoid:
    - Don't use Sonnet or Opus for schedule extraction — Haiku is sufficient and 10x cheaper
    - Don't write to equipment_instances directly — use schedule_extractions staging table
    - Don't modify the existing extract() function in extractor.py
    - Don't process non-schedule drawings in this extractor
