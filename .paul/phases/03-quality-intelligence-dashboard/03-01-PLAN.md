<objective>
## Goal
Create the Quality module web layer — blueprint, dashboard page, and issue browsing — following existing QMS patterns (projects/welding dashboards). This is the first user-facing surface for the quality_issues data imported in Phase 2.

## Purpose
Phases 1-2 built the data foundation (schema + import engine + 534 tests). Without a web UI, users must use CLI commands to see quality data. This plan delivers the dashboard and issue list that makes imported Procore observations immediately useful to the team.

## Output
- `api/quality.py` — Flask blueprint with dashboard route, issue list API, stats API
- `frontend/templates/quality/dashboard.html` — Dashboard with stats cards, recent issues, issues-by-project breakdown, quick actions
- Module registered in config.yaml and nav tabs
</objective>

<acceptance_criteria>

## AC-1: Quality Module Accessible via Nav
```gherkin
Given a logged-in user with quality module access
When they view any QMS page
Then a "Quality" tab appears in the navigation bar
And clicking it navigates to /quality/
```

## AC-2: Dashboard Stats Display
```gherkin
Given quality_issues data exists in the database
When a user visits /quality/
Then they see stats cards showing: Total Issues, Open Issues, In Progress, Critical Issues
And the counts match the actual database values
And critical issues card has warning styling when count > 0
```

## AC-3: Recent Issues Table
```gherkin
Given quality_issues data exists
When a user views the dashboard
Then they see the 10 most recent issues in a table
And each row shows: title, type badge, project, status badge, severity, created date
And the table handles empty state gracefully (no issues imported yet)
```

## AC-4: Issues by Project Breakdown
```gherkin
Given quality_issues linked to projects exist
When a user views the dashboard
Then they see a project breakdown showing: project number/name, total issues, open issues, critical count
And projects are sorted by total issues descending
```

## AC-5: Business Unit Filtering
```gherkin
Given a non-admin user with restricted business unit access
When they view the quality dashboard
Then stats, recent issues, and project breakdown only show data from their assigned business units
And admin users see all data regardless
```

## AC-6: Issue List API
```gherkin
Given quality_issues data exists
When a client requests GET /quality/api/issues
Then it returns a JSON array of issues with project info joined
And supports optional query parameters: type, status, severity, project_id
And respects business unit filtering
```

<task type="auto">
  <name>Task 1: Create quality blueprint with dashboard and API routes</name>
  <files>api/quality.py, api/__init__.py, config.yaml</files>
  <action>
    Create `api/quality.py` as a Flask blueprint following the projects/welding pattern:

    **Blueprint setup:**
    - `bp = Blueprint("quality", __name__, url_prefix="/quality", template_folder="../frontend/templates")`

    **Helper: `_user_bu_ids()`** — copy pattern from api/projects.py:
    - Admin → None (unrestricted), non-admin → session["user"]["business_units"]

    **Helper: `_get_dashboard_stats(conn, bu_ids=None)`** — returns dict with:
    - total_issues, open_issues, in_progress (in_progress + in_review), critical_issues (critical + not closed)
    - Apply BU filter via `business_unit_id IN (...)` when bu_ids is not None
    - Handle the WHERE/AND clause correctly (use base WHERE 1=1 pattern)

    **Helper: `_get_recent_issues(conn, limit=10, bu_ids=None)`** — returns list of dicts:
    - JOIN projects on project_id for project_number, project_name
    - ORDER BY created_at DESC, LIMIT
    - Apply BU filter

    **Helper: `_get_issues_by_project(conn, bu_ids=None)`** — returns list of dicts:
    - GROUP BY project_id with counts: total, open, critical
    - JOIN projects for number/name
    - ORDER BY total DESC

    **Route: `@bp.route("/")`** → `dashboard()`:
    - Call stats, recent, by_project helpers
    - render_template("quality/dashboard.html", stats=stats, recent=recent, by_project=by_project)

    **Route: `@bp.route("/api/issues")`** → `api_issues()`:
    - Accept query params: type, status, severity, project_id (all optional filters)
    - JOIN projects, LEFT JOIN root_causes
    - Apply BU filter
    - Return jsonify(issues)

    **Route: `@bp.route("/api/stats")`** → `api_stats()`:
    - Return jsonify of dashboard stats

    **Register in api/__init__.py:**
    - Import and register blueprint: `from qms.api.quality import bp as quality_bp`
    - Add to _BLUEPRINT_MODULE mapping: `"quality": "quality"`

    **Register in config.yaml:**
    - Add to web_modules section:
      ```yaml
      quality:
        label: "Quality"
        default_endpoint: "quality.dashboard"
      ```
    - Place it after "projects" in the YAML ordering for logical nav position

    **Avoid:**
    - Do NOT modify quality/db.py or quality/schema.sql — schema is locked
    - Do NOT add new dependencies
    - Use `get_db(readonly=True)` for all read operations (existing pattern)
  </action>
  <verify>
    python -c "from qms.api.quality import bp; print(bp.name, bp.url_prefix)"
    → Should print: quality /quality
  </verify>
