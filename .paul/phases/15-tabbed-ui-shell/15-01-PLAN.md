<objective>
## Goal
Refactor the single-page quality manual viewer into a 4-tab UI shell (M1, M2, M3 Programs, M4 SOPs) with per-tab TOC filtering, cross-module search, and empty-state placeholders for M3/M4.

## Purpose
The existing viewer combines M1+M2 in one undifferentiated view with no path to M3/M4 content. This plan creates the navigation structure that all subsequent phases (16-18) plug into.

## Output
- Refactored `frontend/templates/qualitydocs/index.html` with horizontal tab bar
- M1 and M2 as separate tabs with filtered TOC
- M3 Programs and M4 SOPs tabs with empty-state UI
- Cross-module search spanning M1/M2 content + M3/M4 APIs
- URL hash routing for tab persistence
</objective>

## AC-1: Tab Navigation
```gherkin
Given the quality manual page loads at /qualitydocs/
When the page renders
Then a horizontal tab bar shows 4 tabs: "M1 — Quality Manual", "M2 — Compliance", "M3 — Programs", "M4 — SOPs"
And the first tab (M1) is active by default
And clicking a tab switches the visible content area
```

## AC-2: M1/M2 Separated Views
```gherkin
Given the M1 tab is active
When the TOC renders
Then only Module 1 sections appear in the sidebar
And clicking a section loads its content in the content area

Given the M2 tab is active
When the TOC renders
Then only Module 2 sections appear in the sidebar
```

## AC-3: M3/M4 Empty States
```gherkin
Given the M3 Programs tab is active
When the tab content renders
Then an empty-state message displays: "Programs — Coming Soon" with brief description
And the TOC sidebar is hidden or shows a placeholder

Given the M4 SOPs tab is active
When the tab content renders
Then an empty-state message displays: "SOP Catalog — Coming Soon" with brief description
```

## AC-4: Cross-Module Search
```gherkin
Given any tab is active
When the user types a query in the search bar and submits
Then search results from M1/M2 FTS5 are displayed
And results indicate which module each result belongs to
And clicking a result switches to the correct tab and loads the section
```

## AC-5: URL Hash Routing
```gherkin
Given the user clicks the M2 tab
When the URL updates
Then the hash changes to #m2
And refreshing the page with #m2 in the URL restores M2 as the active tab
```

## AC-6: Stats Cards Update
```gherkin
Given the M1 tab is active
When stats render
Then stats show M1-specific counts (sections, content blocks, cross-refs)

Given the M2 tab is active
When stats render
Then stats show M2-specific counts
```

    - Do NOT add new dependencies or external libraries
