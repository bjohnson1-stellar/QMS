<objective>
## Goal
Build cross-discipline conflict detection engine and negative space scanner for the equipment registry. Detect attribute mismatches between disciplines (hp, voltage, amperage, weight, pipe_size) using the configurable conflict_rules table, identify missing discipline coverage per equipment type, classify severity, and generate RFI-ready conflict reports.

## Purpose
The equipment registry (Phase 19) unified 585 equipment records across disciplines, but doesn't yet detect problems. This phase adds the "intelligence" — automatically finding where drawings disagree with each other (attribute conflicts) and where disciplines haven't documented equipment they should own (negative space). These are the two highest-value outputs for construction QA: catching coordination issues before they become field problems.

## Output
- `pipeline/conflict_detector.py` — detection engine with attribute comparison and negative space scanning
- Enhanced `pipeline/reconciler.py` — captures per-discipline attributes during reconciliation
- `pipeline/cli.py` — `conflicts` CLI command with summary + RFI output
- Populated `equipment_conflicts` table for Vital (07645)
</objective>

<task type="auto">
  <name>Task 2: Build conflict detection engine</name>
  <files>pipeline/conflict_detector.py</files>
  <action>
    Create `pipeline/conflict_detector.py` with two detection passes:

    **Pass 1: Attribute Conflicts**
    - `detect_attribute_conflicts(project_id)`:
      - For each equipment instance with 2+ appearances, compare `attributes_on_sheet` across disciplines
      - Load active conflict_rules from DB
      - For each rule's `attribute_name`, check if the attribute exists in multiple appearances
      - Apply comparison logic based on `comparison_type`:
        - `exact`: flag if values differ (normalize strings: strip, lowercase)
        - `numeric_tolerance`: parse numeric values, flag if difference > tolerance (absolute or percent)
        - `presence`: flag if attribute exists in one discipline but not another
      - Insert into equipment_conflicts with: conflict_type='attribute', both discipline/drawing/value pairs, rule_id, severity
      - Handle unit normalization: "480V" vs "480" vs "480/277V" (extract primary voltage)
      - Handle None/empty gracefully — missing data is not a conflict (that's negative space)

    **Pass 2: Negative Space (Missing Disciplines)**
    - `detect_missing_disciplines(project_id)`:
      - For each equipment instance, get its type's `expected_disciplines`
      - Get actual disciplines from equipment_appearances
      - For each expected discipline not in actual:
        - Create equipment_conflicts row with conflict_type='missing_discipline'
        - discipline_a = expected (missing), value_a = 'not found'
        - discipline_b = primary discipline, value_b = 'present'
        - Severity: 'critical' if the missing discipline is the equipment's primary expected discipline; 'warning' otherwise
      - Skip equipment types with empty expected_disciplines (instruments, unknown)

    **Orchestrator**
    - `detect_conflicts(project_id, clear_existing=True)`:
      - If clear_existing: DELETE FROM equipment_conflicts WHERE project_id = ? AND status = 'new'
      - Run both passes
      - Return DetectionResult dataclass with counts by type and severity

    Avoid: Flagging conflicts for resolved/false_positive status rows (only clear 'new' status).
    Avoid: Creating duplicate conflicts on re-run (clear + re-insert pattern).
  </action>
