<acceptance_criteria>

## AC-1: API Mutation Rate Limiting
```gherkin
Given an authenticated user making rapid API requests
When more than 60 POST/PUT/DELETE requests are made from the same IP within 60 seconds
Then subsequent requests return HTTP 429 with JSON {"error": "Rate limit exceeded. Try again in N seconds."}
And a Retry-After header is included in the response
```

## AC-2: Login Rate Limit Returns 429
```gherkin
Given a user who has exceeded login attempt limits
When they submit another login attempt
Then the response is HTTP 429 (not 200 with error message in HTML body)
And the response includes Retry-After header with seconds remaining
And the login page still renders with the error message for browser users
```

## AC-3: CSRF Coverage on All State-Changing Methods
```gherkin
Given the CSRF protection middleware
When a PUT or DELETE request is made with form-encoded content type
Then CSRF token validation is enforced (same as POST)
And JSON content-type requests remain exempt (SameSite=Lax covers them)
```

## AC-4: Origin Header Validation for JSON APIs
```gherkin
Given a JSON API request (application/json content type)
When the Origin header is present and does not match the server's host
Then the request is rejected with 403
And requests with no Origin header or matching Origin are allowed through
```

## AC-5: No Regressions
```gherkin
Given the existing test suite
When python -m pytest tests/ -x is run
Then all tests pass with no new failures
```

       - Tracks requests per IP (not per IP+email like login)
       - Constants: `MAX_REQUESTS = 60`, `WINDOW_SECONDS = 60` (60 req/min per IP)
       - Same thread-safe pattern as LoginRateLimiter (Lock + monotonic timestamps)

       - Only trigger on POST, PUT, DELETE, PATCH methods (skip GET/HEAD/OPTIONS)
       - Skip `auth.*` endpoints (login has its own limiter) and `static` endpoint
       - Call `api_limiter.record(request.remote_addr)`
       - Call `api_limiter.check(request.remote_addr)` — if blocked, return:
         ```python
         response = jsonify({"error": f"Rate limit exceeded. Try again in {wait} seconds."})
         response.status_code = 429
         response.headers["Retry-After"] = str(wait)
         return response
         ```
       - Import `jsonify` at the top of the file (already imported from flask)

         ```python
         if request.content_type and "json" in request.content_type:

             response.status_code = 429
             response.headers["Retry-After"] = str(wait)
             return response
         return render_template("auth/login.html",
                                error=f"Too many failed attempts. Try again in {wait} seconds."), 429
         ```
       - This gives both JSON API clients and browser users a proper 429

<task type="auto">
  <name>Task 2: CSRF hardening — extend method coverage and add Origin check</name>
  <files>api/__init__.py</files>
  <action>
    1. In api/__init__.py, update `check_csrf()`:
       - Change `if request.method != "POST":` to:
         ```python
         if request.method not in ("POST", "PUT", "DELETE", "PATCH"):
         ```
       - This extends CSRF token validation to all state-changing methods for form submissions
       - JSON content-type requests remain exempt (next step adds Origin check for those)

    2. In api/__init__.py, add Origin header validation for JSON APIs:
       - Inside `check_csrf()`, after the JSON content-type early return, add an Origin check:
         ```python
         if "application/json" in content_type:
             # SameSite=Lax covers most CSRF for JSON, but validate Origin as defense-in-depth
             origin = request.headers.get("Origin")
             if origin:
                 # Allow if Origin matches our host (or is localhost/LAN)
                 allowed = {request.host_url.rstrip("/"), f"http://{request.host}"}
                 if origin not in allowed:
                     abort(403, "Origin mismatch.")
             return None  # No Origin header = same-origin request (browsers don't send Origin for same-origin)
         ```
       - This blocks cross-origin JSON requests even if SameSite cookies somehow fail
       - Requests from same origin (no Origin header or matching Origin) pass through

    Avoid: Requiring CSRF tokens in JSON API bodies (would break the existing JavaScript fetch() pattern across all templates). Do NOT change the session nonce generation logic — it's solid. Do NOT add CSRF to GET/HEAD/OPTIONS.
  </action>
