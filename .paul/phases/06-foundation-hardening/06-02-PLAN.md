<objective>
## Goal
Add server-side pagination to the license list API and input validation/sanitization to all license mutation endpoints.

## Purpose
The license list currently returns all rows in a single response — fine for 50 licenses but will degrade with growth. Mutation endpoints accept arbitrary input without validating state codes, date formats, or field lengths. Both issues must be fixed before building new features that add more data and more API surface.

## Output
- `licenses/db.py` — paginated `list_licenses()`, validation helper
- `api/licenses.py` — paginated API response, validation on all POST/PUT endpoints
</objective>

<acceptance_criteria>

## AC-1: Paginated License List
```gherkin
Given 100+ licenses in the database
When GET /api/licenses?page=2&per_page=25 is called
Then the response contains exactly 25 license rows
And the response includes pagination metadata (total, page, per_page, pages)
And the default per_page is 50 when not specified
```

## AC-2: Pagination Backward Compatibility
```gherkin
Given existing code that calls GET /api/licenses without pagination params
When the request has no page or per_page parameters
Then all results are returned (per_page defaults to 0 = no limit)
And the response structure includes pagination metadata alongside the rows
```

## AC-3: Input Validation on License Create/Update
```gherkin
Given a POST /api/licenses request with invalid data
When state_code is not a valid 2-letter US state/territory code
Or expiration_date is not a valid YYYY-MM-DD date
Or license_number exceeds 100 characters
Then the API returns 400 with a clear error message listing all invalid fields
And the license is NOT created/updated
```

## AC-4: Input Validation on CE Endpoints
```gherkin
Given a POST /api/ce-credits request with invalid data
When hours is negative or exceeds 999
Or completion_date is not a valid YYYY-MM-DD date
Then the API returns 400 with a clear error message
And the CE credit is NOT created
```

## AC-5: No Regressions
```gherkin
Given the existing test suite
When python -m pytest tests/ -x is run
Then all tests pass with no new failures
```

<task type="auto">
  <name>Task 1: Add server-side pagination to license list</name>
  <files>licenses/db.py, api/licenses.py</files>
  <action>
    1. In licenses/db.py, add pagination parameters to `list_licenses()`:
       - Add `page: int = 0` and `per_page: int = 0` keyword args
       - If per_page > 0: add `LIMIT ? OFFSET ?` to the SQL query
       - Add a COUNT query (same WHERE clause, no LIMIT) to get total
       - Return a dict: `{"items": [...], "total": N, "page": page, "per_page": per_page, "pages": ceil(total/per_page)}`
       - If per_page == 0 (default): return all rows in same dict structure with pages=1
       - Cap per_page at 200 maximum to prevent abuse

    2. In api/licenses.py, update `api_list_licenses()`:
       - Parse `page` and `per_page` from request.args (default 0 for both = all results)
       - Convert to int, clamp to valid ranges (page >= 1, 1 <= per_page <= 200, or 0 for no limit)
       - Pass to `list_licenses()`
       - Return the paginated response dict directly (not just the rows)

    3. In api/licenses.py, update `export_licenses_csv()`:
       - Continue using list_licenses() WITHOUT pagination (per_page=0) for full export
       - Extract items from the returned dict: `result["items"]`

    Avoid: Breaking the state_detail_page() which calls list_licenses_for_state() — that function is separate and not paginated. Do NOT add pagination to list_licenses_for_state().
  </action>
  <verify>
    - python -m pytest tests/ -x (no regressions)
    - Manual: GET /api/licenses?page=1&per_page=5 returns 5 items with total count
    - Manual: GET /api/licenses (no params) returns all items
  </verify>
  <done>AC-1, AC-2 satisfied: License list API supports pagination with backward compatibility</done>
</task>

<task type="auto">
  <name>Task 2: Add input validation to all license mutation endpoints</name>
  <files>api/licenses.py</files>
  <action>
    1. In api/licenses.py, add a validation helper `_validate_license_fields(data, require_all=False)`:
       - `state_code`: Must be a key in `_STATE_NAMES` (the existing dict). Normalize to uppercase.
       - `expiration_date`, `issued_date`, `association_date`, `disassociation_date`: If present, must be valid YYYY-MM-DD format (use `datetime.strptime`)
       - `license_number`: If present, strip whitespace, max 100 chars
       - `license_type`: If present, strip whitespace, max 100 chars
       - `holder_type`: If present, must be one of ("company", "employee")
       - `status`: If present, must be one of ("active", "expired", "suspended", "pending")
       - `employee_id`: If present and holder_type is "employee", validate it's a non-empty string
       - Returns list of error strings. Empty list = valid.
       - When `require_all=True` (for create), also checks required fields are present

    2. Add a validation helper `_validate_ce_credit_fields(data)`:
       - `hours`: Must be a positive number, max 999
       - `completion_date`: Must be valid YYYY-MM-DD
       - `course_name`: Strip whitespace, max 200 chars
       - Returns list of error strings.

    3. Add a validation helper `_validate_ce_requirement_fields(data)`:
       - `state_code`: Must be in `_STATE_NAMES`
       - `hours_required`: Must be a positive number
       - `period_months`: Must be a positive integer
       - Returns list of error strings.

    4. Wire validation into existing endpoints:
       - `api_create_license()`: Call `_validate_license_fields(data, require_all=True)` — return 400 if errors
       - `api_update_license()`: Call `_validate_license_fields(data)` — return 400 if errors
       - `api_create_ce_credit()`: Call `_validate_ce_credit_fields(data)` — return 400 if errors
       - `api_update_ce_credit()`: Call `_validate_ce_credit_fields(data)` — return 400 if errors (for fields present)
       - `api_create_ce_requirement()`: Call `_validate_ce_requirement_fields(data)` — return 400 if errors
       - `api_update_ce_requirement()`: Call `_validate_ce_requirement_fields(data)` — return 400 if errors (for fields present)

    Avoid: Adding validation to read-only endpoints or the import pipeline (import has its own validation via ImportSpec). Do NOT validate scopes endpoints (they just take scope_ids array). Do NOT change the DB layer — validation lives in the API layer only.
  </action>
  <verify>
    - python -m pytest tests/ -x (no regressions)
    - Manual: POST /api/licenses with state_code="ZZ" returns 400 with "Invalid state code"
    - Manual: POST /api/licenses with expiration_date="not-a-date" returns 400
  </verify>
  <done>AC-3, AC-4 satisfied: All mutation endpoints validate input before processing</done>
</task>
