<objective>
## Goal
Build a quality issue import engine that parses CSV files containing quality observations, maps and normalizes fields, deduplicates against existing records, and bulk-inserts into the quality_issues schema. Add CLI commands for import with dry-run support.

## Purpose
This is the data ingestion backbone for Phase 2. By building a data-source-agnostic import engine first, we decouple the "how to get data from Procore" question from the "how to load data into QMS" question. The engine works with any CSV/JSON source — Procore manual exports, API responses, or browser-scraped data all flow through the same pipeline.

## Output
- `quality/import_engine.py` — CSV parser, field mapper, normalizer, dedup inserter
- `quality/cli.py` — Typer CLI app with `import-csv` and `summary` commands
- `cli/main.py` — Register quality CLI module
- `tests/test_quality_import.py` — Import engine tests
</objective>

<task type="auto">
  <name>Task 1: Create quality import engine</name>
  <files>quality/import_engine.py</files>
  <action>
    Create `quality/import_engine.py` following the pattern established by `projects/procore_io.py` (parse → validate → normalize → upsert).

    **Core function: `import_quality_csv(conn, csv_path, *, project_id, source="procore", dry_run=False)`**

    Returns a result dict with counts: `issues_created`, `issues_updated`, `issues_skipped`, `errors`, `skipped_details`.

    **CSV column mapping** (case-insensitive header matching):
    - Title/Name → title (required)
    - Type/Observation Type → type (normalized via normalize_type)
    - Description/Body → description
    - Trade/Responsible Contractor → trade (normalized via normalize_trade)
    - Status → status (normalized via normalize_status)
    - Location → location
    - Priority → priority (lowercased, validated against CHECK constraint)
    - Severity → severity (lowercased, validated against CHECK constraint)
    - Assignee/Assigned To → assigned_to (stored as text, not FK resolved — employee matching is Phase 3)
    - Reported By/Created By → reported_by
    - Due Date → due_date (parse common date formats: YYYY-MM-DD, MM/DD/YYYY, M/D/YYYY)
    - ID/Observation ID/Number → source_id (for dedup)
    - URL/Link → source_url
    - Created At/Created Date → created_at (preserve original timestamps when available)

    **Header matching**: Build a map of canonical_name → [aliases]. Use case-insensitive matching. Skip unmapped columns silently.

    **Processing per row:**
    1. Extract fields using column map
    2. Normalize trade, status, type via quality/db.py functions
    3. Validate: skip rows missing title (log to skipped_details)
    4. Dedup check: if source_id is present, check `quality_issues WHERE source=? AND source_id=?`
       - If exists: UPDATE the existing row, increment `issues_updated`
       - If not: INSERT, increment `issues_created`
    5. Set source_project_id from the Procore project config if available
    6. Set source_synced_at to current timestamp

    **Helper: `_auto_map_headers(headers: List[str]) -> Dict[str, str]`**
    Takes raw CSV headers, returns mapping of csv_header → canonical_field_name.
    Use fuzzy matching: strip whitespace, lowercase, check against alias lists.

    **Import the normalization functions from quality.db** — do NOT duplicate the logic.

    Follow existing QMS patterns:
    - Use `get_logger("qms.quality.import")` for logging
    - Use `conn.execute()` with parameterized queries
    - Handle BOM in CSV files (utf-8-sig encoding, matching procore_io.py)
    - Return structured result dict (matching procore_io.py pattern)
  </action>

  </verify>
