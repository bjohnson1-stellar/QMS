<objective>
## Goal
Add batch import capability, Procore project config mapping, and pipeline integration so observation CSVs can be imported across multiple projects — either manually via CLI batch command or automatically when dropped in the unified inbox.

## Purpose
Plan 02-01 built the single-file import engine. This plan scales it to production use: batch import across projects, auto-detection of observation CSVs in the inbox pipeline, and project-level Procore config mapping. This makes the import pipeline ready for real data — Brandon can export observation CSVs from each Procore project and import them all in one command.

  <name>Task 1: Add batch import engine and project resolver</name>
  <files>quality/import_engine.py</files>
  <action>
    Add to `quality/import_engine.py`:

    **Function: `resolve_project_from_filename(conn, filename: str) -> Optional[Tuple[int, str]]`**
    - Extract project number from filename patterns:
      - "07645 - Observations.csv" → "07645"
      - "Observations_07645.csv" → "07645"
      - "07645_observations_2026-02.csv" → "07645"
      - "07645.csv" → "07645"
    - Use regex: look for 5-digit number pattern (`\b\d{5}\b`) in the filename
    - Look up in projects table: `SELECT id, number FROM projects WHERE number = ?`
    - Return (project_id, project_number) or None if not found

    **Function: `import_batch(conn, directory: str, *, source="procore", dry_run=False, project_number=None) -> Dict`**
    - Scan directory for CSV files (glob *.csv)
    - For each CSV:
      1. If project_number provided, use it for all files
      2. Otherwise, call resolve_project_from_filename()
      3. If no project resolved, add to "unresolved" list and skip
      4. Call import_quality_csv() for each resolved file
    - Return consolidated result:
      ```python
      {
          "files_processed": N,
          "files_skipped": N,
          "total_created": N,
          "total_updated": N,
          "total_skipped": N,
          "per_file": [
              {"file": "07645 - Obs.csv", "project": "07645", "created": N, "updated": N, ...},
          ],
          "unresolved": [
              {"file": "Unknown.csv", "reason": "No project number found in filename"},
          ],
          "errors": [],
      }
      ```
  </action>
  <verify>
    Run: python -c "from qms.quality.import_engine import import_batch, resolve_project_from_filename; print('batch functions load OK')"
  </verify>
  <done>AC-1 satisfied: batch import works. AC-3 satisfied: project resolution from filename.</done>

<task type="auto">
  <name>Task 2: Add CLI batch commands and pipeline config</name>
  <files>quality/cli.py, config.yaml</files>
  <action>
    **quality/cli.py** — Add two new commands:

    **Command: `import-batch`**
    - Arguments: `directory` (required path to directory containing CSVs)
    - Options:
      - `--project`: Optional — if provided, all CSVs map to this project
      - `--source`: Source tag, default "procore"
      - `--dry-run`: Show plan without executing
    - Logic:
      1. Call import_batch(conn, directory, project_number=project, source=source, dry_run=dry_run)
      2. Print per-file results table
      3. Print totals
      4. Print unresolved files (if any)

    **Command: `import-procore`**
    - Arguments: `csv_path` (single CSV file)
    - Options: `--dry-run`
    - Logic: Convenience wrapper that auto-resolves project from filename
      1. Call resolve_project_from_filename()
      2. If found, call import_quality_csv() with resolved project
      3. If not found, print error with suggestion to use --project

    **config.yaml** — Add observation CSV document type to document_types section
    (insert after procore_export):

    ```yaml
    observation_csv:
      patterns:
        - "(?i)^.*observ.*\\.csv$"
        - "(?i)^.*quality.*issues.*\\.csv$"
        - "(?i)^.*deficienc.*\\.csv$"
      destination: "{inbox}/observations"
      handler: "observation-import"
    ```

    This enables the pipeline classifier to auto-detect observation CSVs dropped in inbox.
    The handler tag "observation-import" is for future pipeline routing — no handler code needed yet.
  </action>
  <verify>
    Run: python -m qms.cli.main quality --help
    Verify: Shows import-csv, import-batch, import-procore, and summary commands
  </verify>

<task type="checkpoint:decision" gate="blocking">
  <decision>Procore data extraction method for future work</decision>
  <context>
    Plan 02-01 and 02-02 build the import pipeline that accepts CSV files. The remaining question is HOW to get observation data out of Procore at scale:

    - Each Procore project has an Observations page with export capability
    - API access status is unknown
    - Browser automation (Chrome MCP) is available

    This decision shapes Plan 02-03 and Phase 5 (Procore Push). We need to document the decision NOW so future plans know whether to pursue API integration, browser automation, or manual CSV export.

    **This does NOT block the current plan** — CSV import works regardless of extraction method.
  </context>
  <options>
    <option id="manual-csv">
      <name>Manual CSV Export (for now)</name>
      <pros>Works today, no API credentials needed, Brandon can export from Procore UI manually. Simplest path to getting historical data imported. Can always add automation later.</pros>
      <cons>Manual effort per project per export. Not scalable for ongoing sync. Requires remembering to re-export periodically.</cons>
    </option>
    <option id="browser-automation">
      <name>Browser Automation (Chrome MCP)</name>
      <pros>Fully automated extraction. Chrome MCP tools already available. Can handle login + navigate + export per project.</pros>
      <cons>Fragile — breaks when Procore UI changes. Slow (one project at a time). May trigger rate limits or bot detection. Complex to maintain.</cons>
    </option>
    <option id="investigate-api">
      <name>Investigate Procore API Access</name>
      <pros>Most reliable and scalable. Official supported integration. Batch operations possible. Real-time sync potential.</pros>
      <cons>Requires API credentials (unknown if available). Setup effort. May need Procore admin approval.</cons>
    </option>
  </options>
  <resume-signal>Select: manual-csv, browser-automation, or investigate-api</resume-signal>
